Choosing the Right Agent

The Four Dev Agent Types

Agent TypeBest ForHow to Invoke
ExploreQuick codebase questions: 'where is X defined?', 'what does Y do?'Ask Dispatch to explore the repo
PlanArchitecture decisions before writing any codeAsk Dispatch to plan [feature/change]
Code TaskActual file modifications, bug fixes, new featuresAsk Dispatch to open a code session
General PurposeResearch, documentation, explaining conceptsAny Dispatch message
🧭 Rule of Thumb

Explore first, Plan if needed, then Code. Skipping straight to a code task on an unfamiliar codebase produces worse results.

Git Worktrees and Safety

What Is a Worktree?

A git worktree is an isolated copy of your repository where Claude does all its work. It checks out a new branch automatically, so Claude's changes never touch your main branch until you explicitly merge them.

Without WorktreeWith Worktree
Claude edits files directly on your current branchClaude works on a separate branch in a separate directory
You need to stash or commit your work firstYour work is completely unaffected
Mistakes are hard to undo cleanlyAbandon the worktree and nothing is lost

The Worktree Workflow

  1. You describe the task to Dispatch
  2. Dispatch starts a code session on a new git branch in a worktree
  3. Claude explores, plans, and makes changes in that isolated environment
  4. Claude runs tests (if available) and reports back
  5. You review the diff before deciding to merge
  6. You merge, or ask for revisions, or abandon the branch entirely
🔒 Safety Guarantee

If Claude's code task session is abandoned or produces bad results, zero changes are made to your working tree or main branch.

Prompting Code Tasks Effectively

Anatomy of a Good Code Task Prompt

A well-structured code task prompt has four parts:

  1. What: the specific change or feature to implement
  2. Where: the file(s) or area of the codebase affected
  3. Context: any constraints, patterns to follow, or existing code to be aware of
  4. Verification: how Claude should confirm the change works
✅ Strong Code Task Prompt

What: Add input validation to the user registration endpoint
Where: src/api/auth.py, the register() function
Context: We use Pydantic for validation elsewhere. Follow the same pattern. Raise a 422 with a clear error message on failure.
Verification: Run the existing test suite and confirm no tests fail. If there are no tests for this function, write one.

Common Code Task Patterns

Task TypeHow to Phrase It
Bug fix'Fix the bug where [behaviour] happens when [condition]'
New feature'Add [feature] to [location]. Follow [pattern] from [example file]'
Refactor'Refactor [file/function] to [goal]. Do not change behaviour.'
Test coverage'Write unit tests for [function/class]. Cover happy path, edge cases, errors'
Code review'Review [file] for bugs, code smells, and security issues. Be specific.'

Reviewing Claude's Changes

The Code Review Mindset

Treat every Claude-generated diff as you would a pull request from a junior developer: review it thoroughly before merging.

Review DimensionWhat to Check
CorrectnessDoes the code do what was asked? Are there edge cases it misses?
StyleDoes it match the codebase conventions?
TestsDid Claude write or update tests? Do they cover the important cases?
Side effectsDoes the change affect any other part of the system?
SecurityAre there any new attack surfaces? Are inputs validated?

Asking Claude to Review Its Own Work

  • 'Review the changes you made. What could go wrong?'
  • 'Are there any edge cases your implementation does not handle?'
  • 'Does this code follow the patterns used elsewhere in the codebase?'

Testing and CI Integration

Telling Claude About Your Test Setup

Claude will run whatever test commands you tell it to. Be explicit:

  • 'Run tests with: pytest tests/ -v'
  • 'Run the linter with: flake8 src/'
  • 'Build with: npm run build'
✅ Best Practice

Include your test command in every code task prompt. 'Implement X. Run the test suite with [command] and fix any failures before reporting back.' This forces verification before Claude delivers the result.

Building a Sustainable Dev Workflow

The Recommended Workflow

  1. Explore: understand the relevant area of the codebase
  2. Plan: get an implementation plan for non-trivial changes
  3. Code: run the code task with a well-specified prompt
  4. Verify: review the diff and ask Claude to critique its own work
  5. Test: confirm tests pass
  6. Merge: bring the changes into your main branch

What to Delegate vs. Keep

Delegate to ClaudeKeep for Yourself
Boilerplate and scaffoldingArchitecture decisions affecting the whole system
Test writing for existing codeDefining what counts as correct behaviour
Documentation and docstringsSecurity and privacy reviews
Repetitive refactoringCode that touches sensitive data or auth
Debugging obscure errorsFinal merge and deployment decisions
Practice

Exercises

Exercise 1

Exercise 1 — Explore a Repo (10 min)

Ask Dispatch to run an Explore agent on one of your repos. Ask: 'What are the main entry points, key modules, and any open questions about the architecture?' Review the output for accuracy.

Exercise 2

Exercise 2 — Plan Before Coding (15 min)

Pick a small feature you have been meaning to add. Ask the Plan agent to design an implementation approach. Review the plan: does it match how you would approach it?

Exercise 3

Exercise 3 — Run a Code Task (20 min)

Using the strong prompt template from Module 3, start a code task on a real issue or small feature. Include your test command. Review the diff before merging.

Exercise 4

Exercise 4 — Self-Review Loop (15 min)

After the code task completes, ask Claude: 'Review your changes. What could go wrong, and what edge cases are missing?' Apply any identified fixes.

Key Takeaways

  • Use Explore and Plan before coding on unfamiliar code.
  • Worktrees keep your main branch safe — always.
  • Good code task prompts include what, where, context, and how to verify.
  • Review every diff like a PR from a junior dev.
  • Ask Claude to critique its own work before you review.
  • Test commands belong in every code task prompt.
