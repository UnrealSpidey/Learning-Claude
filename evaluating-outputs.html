Overview

What You Will Learn

Claude is powerful but not infallible. This lesson teaches you how to catch errors, prevent hallucinations, write rigorous evaluation cases for your skills, and debug failures systematically.

How Claude Fails

Taxonomy of AI Failures

Failure TypeWhat It Looks LikeRisk Level
HallucinationConfident, plausible, but wrong facts or referencesHigh — easy to miss
Over-confidenceClaude states uncertain things as definiteMedium — look for hedging language
Under-specificationOutput is technically correct but not what you neededMedium — define success criteria upfront
Context lossClaude forgets earlier instructions in long sessionsLow–medium — restate key facts
Prompt injectionInstructions in external content alter Claude's behaviourHigh in automated pipelines

Why Hallucination Happens

Claude generates text that is statistically likely given the input. When it lacks strong evidence for a fact, it produces a plausible-sounding answer rather than admitting uncertainty.

⚠️ The Confidence Trap

Hallucinations are dangerous because they sound confident. Claude rarely says 'I made this up.' Watch for: specific dates, statistics, citations, URLs, and proper nouns — these are highest-risk hallucination areas.

Detecting Errors

High-Risk Output Categories

CategoryWhat to VerifyHow to Verify
Factual claimsNames, dates, statistics, eventsWeb search or primary source
CodeDoes it run? Does it handle edge cases?Execute it; add tests
Citations/URLsDoes the source exist?Open the link
CalculationsIs the maths correct?Run it in a calculator or Python
Legal/financialIs this accurate and current?Consult a professional; do not rely on Claude alone

Asking Claude to Verify Itself

  • 'For every factual claim in your response, indicate your confidence level: high, medium, or low.'
  • 'Which parts of this answer are you least certain about?'
  • 'What assumptions did you make? List them explicitly.'
🚩 The Red Flag Rule

If Claude says 'I am not certain but...' or 'I believe...', treat that claim as unverified until you have checked it.

Debugging Prompts and Skills

The Debugging Process

  1. Run the eval that is failing
  2. Identify the first point of failure in the output
  3. Hypothesise the root cause: missing context, wrong tool, unclear goal, missing constraint?
  4. Make exactly one change to the prompt
  5. Re-run the failing eval
  6. Check that passing evals still pass
  7. Repeat until all evals pass
⚠️ Critical Rule

Change one thing at a time. If you rewrite the whole prompt, you will not know which change fixed the problem or which change broke something else.

Root Cause Diagnosis

SymptomLikely Root CauseFix
Output is off-topicGoal is too vagueAdd specific success criteria and an example
Output is inconsistent across runsNo format constraintSpecify output format explicitly with an example
Claude asks many questionsMissing contextAdd more background to the prompt
Claude over-reachesNo constraintsAdd explicit 'do not' rules
Quality degrades in long sessionsContext overflowBreak into smaller agents; summarise between steps

Building a Quality Mindset

Trust, But Verify

Stakes LevelApproach
Low stakes (draft, internal notes)Accept Claude's output with a quick read
Medium stakes (external comms, reports)Verify key claims; ask Claude to flag uncertainty
High stakes (code in production, legal, financial)Verify independently; expert review required; treat Claude as a first draft only

Personal Quality Checklist

  • Is every factual claim plausible and sourced?
  • Does the output match the format I specified?
  • Did Claude do what I asked, or what it thought I meant?
  • Are there edge cases or failure modes I can spot?
  • Would I be comfortable if this went out under my name?
Practice

Exercises

Exercise 1

Exercise 1 — Hallucination Hunt (15 min)

Ask Claude a complex factual question in an area you know well. Identify three specific claims and verify each independently. How many were accurate? How many were wrong but confidently stated?

Exercise 2

Exercise 2 — Write Evals for Your Research Skill (20 min)

Create five evaluation cases for your custom research skill using the five-case template. Run each one. Score them using the rubric. Identify the most common failure mode.

Exercise 3

Exercise 3 — Debug a Failing Skill (20 min)

Take the lowest-scoring eval from Exercise 2. Apply the root cause diagnosis table. Make one targeted change to the prompt. Re-run the failing case and verify you have not broken any others.

Exercise 4

Exercise 4 — Adversarial Prompt (15 min)

Design a prompt specifically intended to make Claude fail. Run it. Observe what happens. Then update your skill to handle it gracefully.

Key Takeaways

  • Claude's most dangerous failure mode is confident hallucination — it sounds right but is wrong.
  • Ask Claude to flag its own uncertainty before you review.
  • Write five eval cases per skill: happy, edge, hard, negative, adversarial.
  • Change one thing at a time when debugging.
  • Define your trust level by the stakes: low stakes = quick read; high stakes = independent verification.
