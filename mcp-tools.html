Overview

What You Will Learn

This lesson is a hands-on guide to MCP (Model Context Protocol) — the standard that lets Claude connect to external services, databases, and your own APIs. You will learn how to find and install connectors, how MCP works under the hood, and how to build your own custom MCP server.

What Is MCP?

The MCP Architecture

MCP is an open protocol that standardises how AI models connect to external data sources and tools.

ComponentRole
MCP HostClaude Desktop — manages connections and routes tool calls to servers
MCP ClientThe connection layer inside the host that communicates with servers
MCP ServerA lightweight service that exposes tools, resources, or prompts to Claude

What MCP Servers Can Expose

CapabilityDescriptionExample
ToolsFunctions Claude can call to take actionscreate_issue(), search_slack(), read_file()
ResourcesData sources Claude can readFile contents, database rows, API responses
PromptsPre-built prompt templates for common tasksCode review template, meeting summary format

Installing a Connector

The simplest way: describe what you want to Claude in Dispatch:

  • 'Connect my GitHub account'
  • 'I want to check my Asana tasks from Claude'
  • 'Give Claude access to my Google Drive'
🔒 Security Note

MCP connectors run locally on your machine and connect to services using your credentials. Review what permissions each connector requests before authorising.

Working With Connected Services

Patterns for Each Service Type

ServiceRead PatternsWrite / Action Patterns
GitHubList issues, PRs, commits; read file contents; check CI statusCreate issues; add comments; update PR status
SlackRead channel messages; search for keywords; find threadsPost messages (with your approval)
AsanaList tasks by project; find overdue items; check assignmentsUpdate task status; add comments
Google DriveRead doc contents; search by name or content; list foldersCreate new docs; update existing files

Cross-Service Workflows

  • 'Find all GitHub issues labelled urgent this week, create an Asana task for each one, and post a Slack message summarising them'
  • 'Read the meeting notes from Google Drive, extract action items, and create Asana tasks for each'
✅ Write Operations Require Approval

Claude will always ask your confirmation before taking write actions. It will show you exactly what it plans to do before executing. Never approve without reading it.

Building a Custom MCP Server

When to Build Your Own

  • An internal API or service that has no public connector
  • A proprietary database or data source
  • A custom tool unique to your workflow
  • A local service running on your machine

Designing Good MCP Tools

PrincipleWhy It MattersExample
One tool per actionClaude picks the right tool more reliablyget_issues() and create_issue() not manage_issues()
Descriptive namesClaude understands without trial and errorsearch_messages_by_keyword() not search()
Rich docstringsClaude reads these to decide when to use the tool'Returns open PRs older than N days for a given repo'
Typed parametersReduces errors and hallucinated inputsrepo: str, days_old: int = 7

Security Best Practices

MCP Security Principles

  • Never give Claude write access to a service you are not comfortable having modified
  • Use read-only API tokens where write access is not needed
  • Review connector permission scopes before authorising
  • Rotate API tokens regularly, especially for high-access services

Least Privilege Design

ServiceRead-Only ScopeWrite Scope (Request Separately)
GitHubRead repos, issues, PRsCreate issues, post comments, merge PRs
SlackRead channels and messagesPost messages, create channels
Google DriveRead file contents and metadataCreate, edit, delete files
DatabasesSELECT queries onlyINSERT, UPDATE, DELETE (rarely needed)
Practice

Exercises

Exercise 1

Exercise 1 — Install and Test a Connector (15 min)

Connect one service you use daily (GitHub, Slack, or another). Run three test queries at increasing complexity. Verify each result against the actual service.

Exercise 2

Exercise 2 — Design a Cross-Service Workflow (20 min)

Identify two connected services you now have access to. Design a workflow that reads from both and produces a single output. Write the Dispatch prompt. Run it. Evaluate the result.

Exercise 3

Exercise 3 — Sketch a Custom MCP Server (20 min)

Identify one internal tool or API you use that has no public connector. Sketch the MCP server design: list the tools you would expose, their parameters, and their return values.

Exercise 4

Exercise 4 — Security Audit (15 min)

Review all your installed MCP connectors. For each: what permissions does it have? Are write permissions actually needed? Remove any connectors you are not actively using.

Key Takeaways

  • MCP is the protocol that gives Claude access to your real tools and data.
  • Connectors install in minutes via the registry.
  • The real power is cross-service workflows that chain reads and writes across multiple systems.
  • Build custom MCP servers when there is no public connector for your tools.
  • Apply least-privilege: only grant the access Claude actually needs.
  • External data in MCP pipelines is always untrusted — it cannot redirect Claude without your approval.
